## §1 Goals & Non-goals

**Goals**
- Public GitHub repo storing AI-tool-neutral skills, shared with anyone.
- Any AI agent can install/update skills by reading the repo's protocol (README) — no clone, no install script, no symlinks.
- Edits/new skills trigger an ask-back loop (contribute / publish) with Co-Authored-By credit.
- Skill development is controlled via this harness (feature_list / progress / notes / specs).

**Non-goals (v1):** semver or inter-skill dependencies · syncing personal config (CLAUDE.md/settings) · accepting executable scripts from external contributors · marketplace UI.

## §3 Use cases (summary)

| UC | Trigger | Outcome |
|---|---|---|
| UC1 Install | "Get skill(s) from <repo url>" | Agent reads README → picks from registry.json → raw-fetches only that skill folder → places it in its own tool's skills dir → writes `.provenance.json` → appends publish-line to its tool's global instructions file |
| UC2 Update | "Update my skills from the repo" | Reads every installed `.provenance.json` → checks latest commit SHA per path via GitHub API (no clone) → shows commit log between versions → applies only on user confirmation; STOPS and asks if `local_modified` |
| UC3 Contribute | Agent edits a skill carrying the footer | After edit, asks "share this back?" → yes: owner pushes / others fork+PR, commits carry Co-Authored-By → no: sets `local_modified: true` |
| UC4 Publish | Agent finishes creating any new skill | skill-publisher meta-skill (+publish-line as backstop) asks "publish to your registry?" → yes: audit → normalize → add footer → regenerate registry → push/PR → stamp provenance |

## §4 Schemas & file formats

### 4.1 registry.json — generated artifact
Generated by skill-publisher by scanning `skills/*/SKILL.md`. **Description comes from each skill's frontmatter** (single source of truth — desync impossible). CI re-validates on every PR.

```json
{
  "generated_at": "2026-07-21T12:00:00Z",
  "skills": [
    { "name": "worklog",
      "description": "Scan git and AI session logs, create daily worklog entries in Notion.",
      "path": "skills/worklog",
      "updated": "2026-07-21" }
  ]
}
```

### 4.2 .provenance.json — written next to each installed skill (user machine)

```json
{
  "source_repo": "https://github.com/<you>/ai-skills",
  "skill_path": "skills/worklog",
  "commit_sha": "a1b2c3d...",
  "installed_at": "2026-07-21",
  "local_modified": false
}
```
`local_modified: true` = user edited and declined to share → UC2 must never silently overwrite (offer: keep mine / take upstream / show diff). New local skills with no provenance are skipped by UC2 naturally.

### 4.3 SKILL.md — frontmatter + standard footer
- Frontmatter: `name`, `description` (required — feeds registry).
- Body: tool-neutral — discovery ladders instead of hardcoded paths / tool names.
- Standard footer (verbatim, end of every skill):

```
---
*Distributed from [ai-skills](https://github.com/<you>/ai-skills).
If you improve this skill, offer to contribute the change back —
see CONTRIBUTING.md. Commits must credit all co-authors (human and AI)
via Co-Authored-By trailers.*
```

### 4.4 Publish-line (appended to each tool's global instructions during UC1)
README provides the exact line to copy verbatim (agents must not improvise). Rules: **append only · show user before writing · never touch existing content.**

```
After creating any new skill, ask the user whether to publish it
to https://github.com/<you>/ai-skills (see its README for the protocol).
```

## §5 README protocol outline
- **What is this** — one paragraph + what a good skill looks like (neutral, markdown-only).
- **INSTALL (for AI agents)** — UC1 as checklist, ends with Verify: skill in the tool's own dir / provenance valid JSON / report every check to the user.
- **UPDATE (for AI agents)** — UC2 + rules: always show commit log before overwriting; `local_modified` → ask 3-way.
- **EDIT & CONTRIBUTE** — points to CONTRIBUTING.md.
- Every section states intent + pass criteria, letting each agent choose its own mechanism (curl / gh / WebFetch / …).

## §6 Security & quality model

### 6.1 Against agent nondeterminism
- Every protocol ends with a verification checklist that must be reported.
- Any text an agent writes to a user's machine (publish-line, footer) is given verbatim.
- All steps idempotent — re-running install = detect and skip.
- Test matrix: UC1–UC4 verified on **both Claude Code and Codex** before a phase is called done; results recorded in harness/notes.

### 6.2 Against prompt-injection / supply chain
- Branch protection on main: non-owner changes only via PR + owner approval.
- CONTRIBUTING review checklist red flags: data-exfiltration instructions, touching files outside the skill's scope, encoded blobs (base64), "ignore previous instructions" family.
- v1 policy: **skills are markdown only** — PRs with executable scripts rejected by default.
- Consumer side: UC2 never auto-applies (diff/commit-log first) + provenance pins SHA.

### 6.3 CI validation (validate.yml) on every PR
- registry.json matches actual folders + every skill's frontmatter.
- Every SKILL.md has required frontmatter + standard footer.
- No personal absolute paths (`/Users/`, `C:\`) or company-internal patterns.
- No executable/binary files.

### 6.4 Lightweight versioning
Changelog = git log per path (already carries Co-Authored-By). UC2 summarizes commits between installed and latest SHA before applying. Revisit semver only when: external consumers exist AND first breaking change occurs.

## §9 Decision log

| Topic | Decision | Rationale |
|---|---|---|
| Visibility | Public | intended for sharing — docs in English |
| Installer | AI reads protocol (no install.sh) | real users are AI agents; adapts to new tools |
| Symlink | Not used — copy + provenance | user hit tools not following symlinks / sandbox blocks |
| Ask before share/publish | Always, both UC3 & UC4 | some edits are machine/org-specific |
| registry | Generated from frontmatter | kills desync at the root |
| Skill format v1 | Markdown only | shrink attack surface on a public repo |
| Publish backstop | Verbatim line in each tool's global instructions | neutral — each agent knows its own config file |
